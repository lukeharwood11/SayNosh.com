Google Places API Integration

Uses the Places API (New) — Nearby Search endpoint. Called once on session creation and cached in Supabase for the life of the session.

POST https://places.googleapis.com/v1/places:searchNearby { "includedTypes": ["restaurant"], "maxResultCount": 20, "locationRestriction": { "circle": { "center": { "latitude": HOST_LAT, "longitude": HOST_LNG }, "radius": RADIUS_METERS } }, "rankPreference": "POPULARITY" } // Field mask fields to request: // displayName, rating, priceLevel, primaryTypeDisplayName, // photos, formattedAddress, location, regularOpeningHours.openNow
Matching Algorithm
1
Score each restaurant
All active (non-skipped) members' votes are collected. Any single "no" eliminates the restaurant. All "yes" = Strong Match. Mix of "yes" and "neutral" = Soft Match.
2
Determine outcome
One strong match → reveal winner. Multiple matches → offer "Pick for us" (highest rated) or "Round 2" (re-swipe only the matches). No matches → fall back to highest-rated neutral, or top overall.
3
Round 2 (if chosen)
Matched restaurants are re-inserted with round=2. All members reset has_submitted to false. Same swipe + match pipeline runs again. Round 2 with 1 result skips directly to reveal.
4
Session ends → save friends
save-known-users Edge Function runs, upserting all participant pairs into known_users. session_count increments on subsequent shared sessions.

Edge Cases
Solo sessions → treat as single-voter yes Session timeout → 24h auto-expire Host skips all → minimum quorum = host Round 2 with 1 result → skip choice, go to reveal Round 2 with 0 results → use best R1 match Location denied → manual address entry All cards swiped early → remaining treated as neutral Guest in known_users → prompt to sign up for re-invite Late join → block after swiping starts Places quota → cache results, never re-fetch
